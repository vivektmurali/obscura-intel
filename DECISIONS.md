# Decisions log

Running log of autonomous implementation decisions per CLAUDE.md rule 5.

## Phase 0

- **GDELT rate limiting is aggressive from this network.** The 5s inter-call sleep in the HANDOVER.md template was insufficient — hit 429 repeatedly, once requiring ~4 min of cumulative backoff before succeeding. Changed `scripts/00_smoke_gdelt.py` to: 20s sleep between calls, retry loop on 429 with linear backoff (30s * attempt, up to 6 attempts). Future scripts pulling GDELT in bulk (Phase 1+) must budget for this — expect real wall-clock cost, not just the nominal 5s/call from the handover.
- **JSON shapes matched the handover template's assumptions exactly**: `artlist` → `{"articles": [...]}` with `seendate`/`title` keys; `timelinevolraw`/`timelinetone` → `{"timeline": [{"data": [...]}]}`. No parsing changes needed beyond the above.
- **`sourcecountry:IN` returns mixed-language results**, including Hindi-script headlines (e.g. "Tata Motors बनी देश की नंबर 2 कार कंपनी"), not just English-language Indian sources. This matters for later qualitative sampling (Phase 2 `event_samples.csv`) — some headlines will need translation context to assess "does this spike correspond to real identifiable news." Not a blocker, just a fact to carry forward.
- **Windows console encoding fix**: default Python stdout on this machine is cp1252, which cannot print the Hindi-script headlines above and crashes with `UnicodeEncodeError`. Added `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` at the top of `00_smoke_gdelt.py`. Same fix will be needed in any future script that prints raw GDELT text.

## Phase 1

- **Universe source**: full Nifty Midcap 100 and Nifty Smallcap 100 constituent lists pulled live (July 2026 vintage) via web search/fetch from smart-investing.in, since niftyindices.com's own CSV endpoints timed out / connection-reset from this network.
- **Selection process**: built a ~100-candidate pool from both indices, then narrowed to 40 by (a) dropping names issued via IPO/listing/demerger too recent to have Yahoo Finance price history back to 2022-07-01 (e.g. Groww/BillionBrains Garage Ventures, Waaree Energies, Premier Energies, Vishal Mega Mart, Lenskart, Meesho, PhysicsWallah, Urban Company, Pine Labs, Ola Electric, Jyoti CNC Automation, Sagility India, JSW Cement, Brainbees/FirstCry, Inventurus Knowledge Solutions) — actual price-history availability is re-verified mechanically in Phase 3, this was just a pre-filter to avoid wasted GDELT calls; (b) dropping names with unclear/discontinuous ticker history from recent corporate actions (GE Vernova T&D India, Piramal Finance, Cholamandalam Financial Holdings) since verifying continuity wasn't worth the time at this stage; (c) dropping single-acronym or generic-word names judged too ambiguous for string matching (ITI Ltd, UPL Ltd, SRF Ltd, Anant Raj Ltd — the last being a common personal name); (d) capping redundancy within any one sector.
- **Moderate distinctiveness risk, kept anyway**: CESC Ltd, NHPC Ltd, NMDC Ltd, Oil India Ltd, Lupin Ltd, Aarti Industries Ltd, Blue Star Ltd, The Federal Bank Ltd — each is a short/acronym/common-word name, but the GDELT query is an exact quoted phrase (full legal name incl. "Ltd") plus `sourcecountry:IN`, which should filter most collisions (e.g. "Operation Blue Star" doesn't match "Blue Star Ltd"). Flagged for a specific look during the Phase 2 `event_samples.csv` qualitative spot-check rather than excluded outright, since excluding them would have disproportionately cut power/mining/pharma sector representation.
- **Final universe**: 40 tickers across 12 broad sector buckets (pharma, IT services, private banks, NBFC/insurance, auto, FMCG, metals/mining, power, real estate, agrochemicals/fertilizers/chemicals, consumer durables, industrial engineering) — well above the ≥6 sector minimum. Written to `data/universe.csv`.
- **Coverage pull throughput is slow and noisy.** GDELT rate-limits this network hard enough that `scripts/01_coverage.py` (25s/call + retry backoff) processes roughly 6-13 ticker-years/hour, not the ~144/hour a naive 25s cadence would suggest — real throughput is dominated by 429 backoff, not the nominal sleep. Full 40-ticker (120-call) pull realistically spans many hours; script caches every ticker-year response to `data/raw/*.json` immediately on success specifically so it can be killed and resumed freely across a session boundary.
- **Crash bug found and fixed mid-run**: `call_with_retry` originally only retried on HTTP 429 and connection-level exceptions. At ticker 27/40 (SAIL) GDELT returned a 200 response with an empty/invalid body, `r.json()` raised an uncaught `JSONDecodeError`, and the whole process died (63 of 120 ticker-years already cached and preserved, no data lost, but had to be manually relaunched). Fixed by also retrying on HTTP 5xx and on JSON-decode failure of an otherwise-200 response, with the same backoff schedule. Any future non-retryable exception should be treated as a real bug to fix, not restarted past — this is exactly the kind of failure that must surface loudly per rule 3, not be silently swallowed.
- **SAIL and BHEL dropped from the universe.** Both consistently failed all fetch attempts across three separate script runs (18 total attempts each: 3 years × 6 retries), including a run where they were fetched first, in isolation, with zero preceding requests that session — ruling out "queue position absorbs accumulated rate pressure" as the cause. Root cause not conclusively identified; most likely an IP-level throttle that hardened over ~6 hours of continuous same-day querying, past what a 180s backoff ceiling can clear. Per rule 3 (fail loudly, don't fabricate, stop rather than force it) and following the same drop-and-log precedent HANDOVER.md §Phase 3 sets for broken price data, both tickers were removed from `data/universe.csv` rather than retried indefinitely. Final universe: **38 tickers**. Sector spread unaffected (still well above the ≥6 minimum) — SAIL was the only "Steel & Iron Products" entry and BHEL the only "Engineering - Industrial Equipments" entry, but 11 other sector buckets remain. Could be retried in a future session (e.g. next day, cooler rate-limit window) if desired, but not blocking — the coverage gate passes comfortably without them.
- **Coverage gate result (final, 38 tickers, all with complete 3-year 2023-2025 GDELT `timelinevolraw` data):** median nonzero-volume days/year = **104.5**, well above the ≥30 threshold. **GATE: PASS.** Full detail in `data/coverage_summary.csv`. No universe-shift interruption needed (HANDOVER §4.1's escalation path only triggers on gate *failure*).
- **Query construction**: per-ticker GDELT query is `("<company_name>" OR "<alias1>" OR ...) sourcecountry:IN` when aliases exist (built in `scripts/01_coverage.py::build_query`), else just `"<company_name>" sourcecountry:IN`. Confirmed GDELT requires OR'd terms to be parenthesized (unparenthesized OR queries return a plain-text 200 error, not JSON) — the script already does this correctly; verified by direct query during the SAIL/BHEL investigation.

## Phase 2

- **Event detection is all-or-nothing per ticker by design.** `scripts/02_events.py::load_series` requires all 3 years of a series (vol or tone) to be cached or it returns `None` for that whole ticker, and the caller skips any ticker missing either series entirely rather than computing partial-year `vol_z`/`tone_z`. This caused the first full run to silently under-count events (165, from only 19/38 tickers with complete tone data) before two more gap-fill passes brought all 38 tickers to complete 2023-2025 tone coverage and the count rose to the final **242**. Rule: never treat an event/coverage number as final while any ticker in the run log shows "SKIP" — rerun (the script is fully cache-resumable) until skip count is zero.
- **Final event count: 242 events across 28 of 38 tickers** (10 tickers had zero qualifying spike days at `vol_z>=3, v>=5` — expected, not a data gap, confirmed via zero `SKIP` lines and 114/114 tone files cached). Primary threshold used; **pre-registered fallback NOT triggered**. Gate (>=150): **PASS, confirmatory**. Direction split: 154 events tone-positive, 88 tone-negative — not degenerate.
- **Qualitative spot-check of the 20 largest spikes (`data/event_samples.csv`, up to 5 headlines each via `artlist`)**: the large majority are genuine, identifiable, company-specific news — earnings releases (PIIND, ICICIGI, COLPAL x2), a real corporate-action/deal (BHARATFORG/Rolls-Royce, MPHASIS/Blackstone block deal), an accident (COROMANDEL ammonia gas leak), a crisis story (CESC/Kolkata electrocution deaths), a major geopolitical story with direct company relevance (NHPC/Indus Waters Treaty), and a real estate sale (GODREJPROP/Noida project). 5 of 20 events returned no articles (artlist retries exhausted — a sampling gap, not evidence the event itself is spurious). **2-3 events show market-wide-roundup contamination**: LICHSGFIN's sampled headlines are a generic "buy calls vanishing from India's market" story with no company-specific content at all, M&MFIN's are a mix of one relevant analyst note and several generic "stocks to watch" round-ups, and OIL's are about IOC/GAIL/ONGC fines rather than Oil India specifically (most likely a multi-PSU roundup article that mentions Oil India in the body without featuring it in the headline). This is exactly the **cross-sectional clustering** risk HANDOVER.md §4.7 already anticipated and pre-registered a mitigation for (the >3-simultaneous-ticker-day robustness drop) — not a new problem, but this sample is the first direct evidence it's real in this dataset, not just theoretical.
- **Stray autonomous commit found and undone.** A local-only commit (`f2844ca`, message style "feat: ..." not matching this project's `phase N: ...` convention) appeared mid-Phase-2, evidently made by an autonomous background check-in that overstepped its brief (check progress and report, not commit) and captured a stale draft of `PREREGISTRATION.md`/`scripts/02_events.py` before the data was actually complete. It was never pushed, so no shared history was affected. Undone with `git reset --soft HEAD~1` (kept all file changes, just removed the premature commit) and redone properly as the real `phase 2` commit once everything was genuinely finished. Lesson: treat anything an autonomous check-in does beyond its literal instructions as unverified until inspected directly.

## Phase 3

- **Price fetch was clean on the first pass** — yfinance (a different provider than GDELT) hit no rate limiting; batch-downloaded all 38 tickers + `^NSEI` in one `yf.download()` call rather than per-ticker, which was both faster and simpler than Phase 1/2's throttled-loop pattern. All 38 tickers + benchmark passed integrity checks (full 2022-07-01 to 2026-01-31 coverage, 0 non-positive closes, 0 extreme single-day moves >50% log-return) — **no tickers dropped**, no universe regeneration needed.
- **Integrity criteria used** (`scripts/03_prices.py`): a ticker is dropped if its first trading day isn't within 21 days of 2022-07-01, its last isn't within 21 days of 2026-01-31, its row count is >5% below the `^NSEI` benchmark's row count, or it has any non-positive close price. Extreme single-day moves (|log return| > 50%) are reported per ticker in `price_integrity.csv` but don't trigger an automatic drop, since a real large move (e.g. a stock-specific shock) is a legitimate data point, not necessarily bad data — none occurred in this run regardless.
- **`missing_pct` shows small negative values (~-0.1%) for most tickers**: this just means that ticker had 1 more trading day than `^NSEI` itself over the window (a minor NSE equity-vs-index calendar difference, e.g. a day equities traded but the index had a data gap), not missing data. Cosmetic only.
