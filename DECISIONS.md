# Decisions log

Running log of autonomous implementation decisions per CLAUDE.md rule 5.

## Phase 0

- **GDELT rate limiting is aggressive from this network.** The 5s inter-call sleep in the HANDOVER.md template was insufficient — hit 429 repeatedly, once requiring ~4 min of cumulative backoff before succeeding. Changed `scripts/00_smoke_gdelt.py` to: 20s sleep between calls, retry loop on 429 with linear backoff (30s * attempt, up to 6 attempts). Future scripts pulling GDELT in bulk (Phase 1+) must budget for this — expect real wall-clock cost, not just the nominal 5s/call from the handover.
- **JSON shapes matched the handover template's assumptions exactly**: `artlist` → `{"articles": [...]}` with `seendate`/`title` keys; `timelinevolraw`/`timelinetone` → `{"timeline": [{"data": [...]}]}`. No parsing changes needed beyond the above.
- **`sourcecountry:IN` returns mixed-language results**, including Hindi-script headlines (e.g. "Tata Motors बनी देश की नंबर 2 कार कंपनी"), not just English-language Indian sources. This matters for later qualitative sampling (Phase 2 `event_samples.csv`) — some headlines will need translation context to assess "does this spike correspond to real identifiable news." Not a blocker, just a fact to carry forward.
- **Windows console encoding fix**: default Python stdout on this machine is cp1252, which cannot print the Hindi-script headlines above and crashes with `UnicodeEncodeError`. Added `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` at the top of `00_smoke_gdelt.py`. Same fix will be needed in any future script that prints raw GDELT text.

## Phase 2

- **Event detection is all-or-nothing per ticker by design.** `scripts/02_events.py::load_series` requires all 3 years of a series (vol or tone) to be cached or it returns `None` for that whole ticker, and the caller skips any ticker missing either series entirely rather than computing partial-year `vol_z`/`tone_z`. This caused the first full run to silently under-count events (165, from only 19/38 tickers with complete tone data) before two more gap-fill passes brought all 38 tickers to complete 2023-2025 tone coverage and the count rose to the final **242**. Rule: never treat an event/coverage number as final while any ticker in the run log shows "SKIP" — rerun (the script is fully cache-resumable) until skip count is zero.
- **Final event count: 242 events across 28 of 38 tickers** (10 tickers had zero qualifying spike days at `vol_z>=3, v>=5` — expected, not a data gap, confirmed via zero `SKIP` lines and 114/114 tone files cached). Primary threshold used; **pre-registered fallback NOT triggered**. Gate (>=150): **PASS, confirmatory**. Direction split: 154 events tone-positive, 88 tone-negative — not degenerate.
- **Qualitative spot-check of the 20 largest spikes (`data/event_samples.csv`, up to 5 headlines each via `artlist`)**: the large majority are genuine, identifiable, company-specific news — earnings releases (PIIND, ICICIGI, COLPAL x2), a real corporate-action/deal (BHARATFORG/Rolls-Royce, MPHASIS/Blackstone block deal), an accident (COROMANDEL ammonia gas leak), a crisis story (CESC/Kolkata electrocution deaths), a major geopolitical story with direct company relevance (NHPC/Indus Waters Treaty), and a real estate sale (GODREJPROP/Noida project). 5 of 20 events returned no articles (artlist retries exhausted — a sampling gap, not evidence the event itself is spurious). **2-3 events show market-wide-roundup contamination**: LICHSGFIN's sampled headlines are a generic "buy calls vanishing from India's market" story with no company-specific content at all, M&MFIN's are a mix of one relevant analyst note and several generic "stocks to watch" round-ups, and OIL's are about IOC/GAIL/ONGC fines rather than Oil India specifically (most likely a multi-PSU roundup article that mentions Oil India in the body without featuring it in the headline). This is exactly the **cross-sectional clustering** risk HANDOVER.md §4.7 already anticipated and pre-registered a mitigation for (the >3-simultaneous-ticker-day robustness drop) — not a new problem, but this sample is the first direct evidence it's real in this dataset, not just theoretical.
- **Stray autonomous commit found and undone.** A local-only commit (`f2844ca`, message style "feat: ..." not matching this project's `phase N: ...` convention) appeared mid-Phase-2, evidently made by an autonomous background check-in that overstepped its brief (check progress and report, not commit) and captured a stale draft of `PREREGISTRATION.md`/`scripts/02_events.py` before the data was actually complete. ## Phase 3

- **Price fetch was clean on the first pass** — yfinance (a different provider than GDELT) hit no rate limiting; batch-downloaded all 38 tickers + `^NSEI` in one `yf.download()` call rather than per-ticker, which was both faster and simpler than Phase 1/2's throttled-loop pattern. All 38 tickers + benchmark passed integrity checks (full 2022-07-01 to 2026-01-31 coverage, 0 non-positive closes, 0 extreme single-day moves >50% log-return) — **no tickers dropped**, no universe regeneration needed.
- **Integrity criteria used** (`scripts/03_prices.py`): a ticker is dropped if its first trading day isn't within 21 days of 2022-07-01, its last isn't within 21 days of 2026-01-31, its row count is >5% below the `^NSEI` benchmark's row count, or it has any non-positive close price. Extreme single-day moves (|log return| > 50%) are reported per ticker in `price_integrity.csv` but don't trigger an automatic drop, since a real large move (e.g. a stock-specific shock) is a legitimate data point, not necessarily bad data — none occurred in this run regardless.
- **`missing_pct` shows small negative values (~-0.1%) for most tickers**: this just means that ticker had 1 more trading day than `^NSEI` itself over the window (a minor NSE equity-vs-index calendar difference, e.g. a day equities traded but the index had a data gap), not missing data. Cosmetic only.

## Phase 4

- **Event study ran clean**: all 242 events got a full 120-observation estimation window (market-model beta fit for every event, zero market-adjusted fallbacks needed) — consistent with Phase 3's clean, gapless price data. Only one NaN: NATIONALUM's 2025-12-31 event (the last day of the study period) lacks 20 forward trading days within the fetched price range (which ends 2026-01-30) to compute CAR_t20 — flagged in the script's own report, not silently dropped or fabricated. CAR_t1/CAR_t5 distributions are centered near zero with plausible spread (see `results/car_by_event.csv`), no signs of a computation bug.
- **Followed HANDOVER.md's own phase numbering, not ARCHITECTURE.md's** (parked, post-v0.1): `scripts/04_event_study.py` (CAR computation) and `scripts/05_inference.py` (permutation null + verdict) are committed as two separate phases (`phase 4` / `phase 5`), matching HANDOVER's original plan. ARCHITECTURE.md's "Phase 4: VALIDATION CORE" language collapses both into one label, which caused a momentary mislabeling in conversation — corrected before committing, no code impact.
- **Permutation engine uses a closed-form rolling OLS**, not a per-event `lstsq` refit, for the ~242,000 pseudo-event recomputations (1,000 draws x ~242 events). Validated to floating-point precision against `04_event_study.py`'s per-event `lstsq` output before trusting it (see script docstring). Runtime: ~35s for the full 1,000-permutation run — no rate limits or background execution needed for this phase, unlike Phases 1-3.
- **Permutation null construction**: for each ticker, real events' `tone_z`/`direction` values are paired with freshly-drawn pseudo-event entry dates (same count per ticker as real events, drawn without replacement from that ticker's trading days in the study period, excluding +/-5 trading days around each of that ticker's *real* event entry dates). This holds the marginal distribution of announced tone fixed while randomizing timing — the standard construction for testing whether the *timing* of real spikes carries information, not just their tone values in the abstract.
- **Tone-tercile spread secondary test required its own null** (not just an observed value) to fit properly into the BH-FDR correction alongside the other five secondary tests — implemented by reusing the same fixed tercile membership (based on real `tone_z`, computed once) applied to each permutation's pseudo-CAR_t5 values, rather than re-deriving tercile assignment per permutation (tone_z never changes across draws, only which dates get paired with it).
- **Robustness pass, as pre-registered (one pass, after primary/secondary already recorded)**: (a) fallback spike threshold — not applicable, primary threshold already cleared the 150-event gate with margin; (b) market-adjusted (beta=1) returns — reported for transparency (p=0.833) but explicitly *not* a like-for-like comparison against the market-model null, since the null wasn't rebuilt under market-adjusted assumptions; (c) drop calendar days with >3 simultaneous tickers — **0 days met that threshold** in this dataset (max observed simultaneity was exactly 3, confirmed directly against `events.csv`), so decluster and non-declustered results are identical.
- **VERDICT: NULL.** Primary test: `mean(sign(tone_z) x CAR_t5) = -0.00151`, one-sided permutation p=0.763 (not significant, and the point estimate is even the *wrong* sign relative to the thesis). No secondary test survives BH-FDR at q=0.10 (best secondary p-value: 0.238, for `abs_CAR_t1_vs_null`). Robustness pass doesn't change the conclusion. Per HANDOVER.md's pre-registered kill criterion (§4.8): "no detectable edge under this specification." This is a full, honest result per the pre-registration — not a failure to fix, and no respecification is permitted now that results have been seen. Full detail: `results/stats.json`.

## Phase 6 (post-v0.1, same override — NULL-mode scoring only)

- **Verdict-branch guard is a hard `sys.exit`, not a comment.** `scripts/06_scoring.py` reads `results/stats.json`'s verdict at runtime and refuses to run (exit 1) if it's ever anything other than `NULL` — SIGNAL-mode scoring (historical CAR(t+5) bucket stats) isn't implemented, and silently falling through to NULL-mode logic on a future SIGNAL verdict would violate the claims-audit gate by under-reporting what validation actually supports. - **Claims-audit gate enforced by assertion, not convention**: `score_null_mode` asserts no `CAR_t*` column is present in the frame before scoring. - **Intensity percentile uses v0.1's locked historical `vol_z` distribution as the reference** (`data/events.csv`, 242 events, audited and fixed), not the ever-growing live event table. - **Novelty is per-ticker days-since-last-event within the live cumulative table** (`data/live/events.csv`), `NaN` for a ticker's first-ever recorded event (no prior event to diff against) rather than 0 or some other placeholder — an explicit "not applicable" beats a fabricated number.

## Post-Phase-8 bug: `parse_timeline` broke on the first true 1-day gap (2026-07-11)

- **Cron run [29139679607](https://github.com/vivektmurali/obscura-intel/actions/runs/29139679607) failed** (`daily-pipeline`, 2026-07-11 04:29 UTC scheduled run, "All jobs have failed") — the first cron run to hit a genuine 1-day incremental gap rather than the multi-month backfill Phases 5-8 were built and tested against. This resets the 14-consecutive-clean-run counter toward `v1.0` back to zero (1 clean run — 2026-07-10 — before this).
- **Root cause, confirmed empirically against the live API (not guessed):** GDELT's `timelinevolraw`/`timelinetone` timeline modes auto-select bucket resolution by query span. Multi-year historical queries return exactly one daily bucket per day (`T000000Z` only — verified against the cached `data/raw/*.json` files). Short-span queries — like the live pipeline's real 1-day gaps — return 15-minute buckets (`T000000Z`, `T001500Z`, `T003000Z`, ... confirmed via a direct API call for a 2-day window). `scripts/10_daily_ingest.py::parse_timeline` hardcoded a midnight-only `"T000000Z"` strip-and-parse, so the first non-midnight bucket timestamp crashed it with an unhandled `ValueError`. The crash happened mid-fetch, before any `to_parquet` write, so no bad data was written — the pipeline's "fail loudly, don't corrupt" design worked as intended, it just needed the actual fix.
- **Fix**: `parse_timeline(data, kind)` now parses the full timestamp (`%Y%m%dT%H%M%SZ`) instead of assuming midnight, then groups by calendar date and aggregates: `sum` for `kind="vol"` (raw article counts — summing sub-daily bucket counts recovers the true daily total), `mean` for `kind="tone"` (unweighted average of per-bucket averages — an approximation, since GDELT doesn't expose per-bucket article counts to weight by; exact when buckets are already daily, i.e. the historical bootstrap path). ## Third production bug: phantom-day rows from GDELT's inclusive end-boundary (2026-07-16)

- **Found during a code review of this session's own work** (the parse_timeline fix, the retry tuning, the test suite, and the power analysis script) — not from a failed cron run. GDELT's `enddatetime` query parameter is inclusive, not exclusive as `fetch_gap`'s own docstring assumes ("start_date inclusive, end_date exclusive"). A query for `[gap_start, target_date+1)` actually returns data through `target_date+1`'s exact midnight — one boundary bucket. After this session's earlier `parse_timeline` fix started successfully grouping sub-daily buckets by calendar date (instead of crashing on them), that single boundary instant became a full — but near-meaningless — row for `target_date+1`. `update_gdelt_series` merges it in, so the store's `date.max()` advances one day past what was actually fetched. The next run's gap check (`gap_start > target_date` -> skip) then treats that day as already covered and never fetches its real data. **Permanently** for that specific day — the pattern oscillates day-to-day (each run "confirms" a new phantom day while never revisiting the previous one), so roughly every other day for an affected ticker silently carried a near-zero artifact instead of a real aggregate.
- **This is a new regression, not a pre-existing bug**: the old (pre-this-session) `parse_timeline` crashed on any non-midnight bucket, so it never lived long enough to reach this code path. It could only start corrupting data from this session's first live post-fix run (2026-07-12) onward.
- **Confirmed live in production before fixing, not just in theory**: `data/live/volume_daily.parquet`'s committed state showed BIOCON frozen at max-date 2026-07-14 (value 0.0), and the 2026-07-15 scheduled cron log shows zero "fetching BIOCON vol" lines at all — silently skipped, proving the stuck state existed before that run even began. 19 of 38 tickers showed this pattern (max date in {07-12, 07-14, 07-15}); the other 19 hadn't had a post-fix fetch yet (still at 07-09) and were unaffected.
- **Fix**: `fetch_gap` now filters `parse_timeline`'s result to `df["date"] < end_date` before returning. This is deliberately fixed in `fetch_gap`, not `parse_timeline` — the "end_date exclusive" contract belongs to the caller that constructs the query range, and `parse_timeline` is shared with the historical bootstrap path, which has no reason to know about this exclusivity convention.
- **Repair**: truncated all rows with `date >= 2026-07-10` from both `data/live/volume_daily.parquet` and `data/live/tone_daily.parquet` (93 and 90 rows respectively, across 19 tickers) — a conservative cutoff before any post-fix run could have executed — then ran the normal (non-`--backfill`) daily ingest. The existing recompute-over-append gap-detection naturally saw the enlarged gap and refetched the whole window fresh with the fixed code; no special-case repair logic was needed. Verified post-repair: zero rows anywhere beyond the run's own `target_date` (2026-07-15), confirming the boundary-exclusion contract now holds against the real live API, not just a fixture.
- **Regression test added** (`tests/test_10_daily_ingest.py::FetchGapBoundaryTests`): mocks `requests.get` to return the exact real fixture that exposed this bug (`live_subdaily_multiday_biocon_vol.json`, spanning 2026-07-10 to 2026-07-14) and asserts `fetch_gap` excludes the 07-14 boundary row. Negative-control validated: the same fixture run through the pre-fix logic returns 5 rows including the phantom day; post-fix, 4. 12 tests total now, all passing.
- **Found by an adversarial code review, not a crash.** Worth noting as a process point: this bug produced no exception, no non-zero exit, no failed cron run — it degraded data quality silently and would likely not have been caught by monitoring alone. The review that found it was explicitly asked for by Vivek as a general code-review pass, not triggered by any observed failure.
